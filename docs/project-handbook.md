# admin-lab 项目协作手册

## 1. 文档目的

这份文档面向后续参与 `admin-lab` 的所有项目成员。

目标只有一件事：让成员在阅读后，能快速理解这个项目为什么存在、应该做什么、不应该做什么、代码应该怎么组织、功能应该怎么落地、协作时应该遵守哪些规则。

`admin-lab` 不是一个普通业务系统，它是团队的外网实验场、业务预演场和可复用资产沉淀库。

## 2. 项目背景

公司实际后台项目长期在内网开发，真实研发过程中会反复遇到这些问题：

- 想验证一个不熟悉的新前端组件，但内网试错成本高
- 想尝试一个新的后端库或实现方式，但不适合直接在正式项目里摸索
- 某些高频业务能力反复出现，例如 Excel 导入导出、可视化、复杂表单、附件处理，但缺少统一沉淀
- 某个真实需求还没有完全想清楚，希望先做一版可运行原型，再搬回内网落地

`admin-lab` 就是为了解决这些问题而建立的。

## 3. 项目定位

`admin-lab` 的定位是：

- 外网实验项目
- 真实需求预演项目
- 可复用组件与模块资产库
- 内网项目的前置验证场

这个项目的核心价值不是承载正式生产业务，而是为正式项目降低试错成本、降低重复开发成本、提高方案验证效率。

## 4. 项目目标

`admin-lab` 长期目标如下：

- 为团队提供一个安全、自由、快速的实验环境
- 将高频业务能力沉淀为可复用的前后端模块
- 让真实需求可以先在这里完成一版仿真开发
- 为内网项目提供可借鉴、可裁剪、可搬运的实现参考
- 逐步形成团队统一的后台模块开发方式

## 5. 项目边界

为了保证项目长期可维护，必须明确边界。

应该做的：

- 对真实业务需求进行仿真开发
- 验证新组件、新库、新交互、新实现方式
- 沉淀高频复用组件、复用模块、复用页面模式
- 输出配套说明，便于后续搬运到内网

不应该做的：

- 不把 `admin-lab` 做成新的正式业务系统
- 不在这里堆积与后台研发无关的展示型页面
- 不引入和团队长期技术路线明显冲突的能力
- 不为了“炫技”加入难以迁移、难以维护的实现
- 不提交真实生产配置、真实账号、真实业务敏感数据

判断一个功能是否适合进入 `admin-lab`，可以先问两个问题：

- 它是否服务于团队未来的真实后台研发？
- 它是否有实验价值、复用价值或迁移价值？

如果两者都没有，就不建议放进来。

## 6. 功能分区

项目当前功能方向按三个区域组织：

### 6.1 需求仿真

作用：用于先模拟公司真实业务需求，快速完成一版可运行页面或流程。

适合放入这里的内容：

- 列表页、详情页、编辑页
- 审批流、状态流页面
- 大屏、看板、统计分析页
- 某个真实业务需求的完整前后端闭环原型

这里强调的是“业务场景预演”。
要求尽量贴近真实需求结构，而不是只验证局部技术点。

### 6.2 组件示例

作用：用于验证不熟悉、未使用过、风险未知的新组件或新交互。

适合放入这里的内容：

- 新表格组件
- 新图表库
- 富文本、拖拽、打印、虚拟列表、地图、二维码等技术示例
- 某个组件的多种用法演示

这里强调的是“技术验证”。
每个示例尽量独立、最小可运行、说明清楚。

### 6.3 复用组件

作用：沉淀在多个项目、多类业务中都可能重复使用的能力。

适合放入这里的内容：

- Excel 导入导出
- 通用搜索栏
- 通用表格能力
- 字典选择器
- 附件上传
- 图表容器
- 批量操作工具条

这里强调的是“可迁移资产”。
不仅要能运行，还要便于搬运、裁剪、扩展。

## 7. 功能归类原则

同一个能力在不同阶段可能位于不同分区，建议按下面的顺序演进：

1. 新技术初次尝试，先进入 `组件示例`
2. 能组合成真实页面或流程后，进入 `需求仿真`
3. 具备跨项目复用价值后，再沉淀到 `复用组件`

这意味着：

- `组件示例` 偏试验
- `需求仿真` 偏场景
- `复用组件` 偏资产

## 8. 技术基线

当前项目基于 BoyKing Admin 的现有裁剪架构，技术基线保持一致：

- 后端：Go + Gin + GORM + MySQL + Casbin
- 前端：Vue 3 + Vite + Pinia + Element Plus

项目不是通用插件平台，也不是新架构试验田。
新能力优先在现有技术边界内落地，避免无节制扩张。

## 9. 前后端目录规划

### 9.1 后端分包

后端统一遵循：

`router -> api -> service -> model`

实验室模块按功能区继续分包：

- `server/api/v1/lab/simulation`
- `server/api/v1/lab/componentdemo`
- `server/api/v1/lab/reusable`
- `server/router/lab/simulation`
- `server/router/lab/componentdemo`
- `server/router/lab/reusable`
- `server/service/lab/simulation`
- `server/service/lab/componentdemo`
- `server/service/lab/reusable`
- `server/model/lab/...`

约定如下：

- `simulation` 放需求仿真后端能力
- `componentdemo` 放组件示例对应接口
- `reusable` 放可复用模块接口和服务

不要把实验室代码继续堆在 `example` 这类含义不清的包里。

### 9.2 前端分包

前端按实验室菜单结构组织：

- `web/src/view/lab/simulation/...`
- `web/src/view/lab/component-demo/...`
- `web/src/view/lab/reusable/...`
- `web/src/api/...`

约定如下：

- 页面放在对应 `view/lab/...` 目录
- 页面接口放到 `web/src/api/`
- 如果某个复用能力未来会被多个页面共享，可再抽到公共组件目录

## 10. 菜单规划约定

当前菜单主结构为：

- `实验室`
- `实验室 / 需求仿真 / 概览`
- `实验室 / 组件示例 / 概览`
- `实验室 / 复用组件 / 概览`

后续新增功能时，遵循下面规则：

- 分类说明页保留为各分区的 `概览`
- 具体能力放在其下一级菜单
- 一个功能只属于一个主分区，避免重复挂载
- 菜单命名尽量直接表达业务或能力，不使用模糊名称

例如：

- `实验室 / 复用组件 / Excel 实验面板`
- `实验室 / 组件示例 / 虚拟列表示例`
- `实验室 / 需求仿真 / 客户档案审批仿真`

## 11. 开发什么

功能选题优先级建议如下：

第一优先级：

- 高概率在内网项目复用的能力
- 当前团队正在频繁遇到的业务痛点
- 明显能节省重复开发成本的模块

第二优先级：

- 某类新技术或新组件的验证型页面
- 尚未确定方案但需要预演的真实需求

第三优先级：

- 用于补齐实验室示例体系的辅助页面

不建议优先开发：

- 只适用于单个极窄业务的重定制页面
- 没有迁移价值的演示内容
- 无明确使用场景的“纯收藏式”功能

## 12. 功能准入标准

新功能进入 `admin-lab` 之前，至少要满足下面几项中的大部分：

- 有明确背景，知道为什么做
- 有清晰分类，知道属于哪个分区
- 有最小可运行目标
- 有后续可能迁移到内网的预期
- 有配套说明，别人能看懂怎么用

如果一个功能只是临时测试且没有沉淀价值，可以本地试，不一定要进仓库。

## 13. 开发规范

### 13.1 通用原则

- 先保证可理解，再追求复杂能力
- 先做最小闭环，再考虑扩展
- 优先和现有脚手架保持一致
- 每个实验模块都应尽量具备说明、页面、接口、示例数据或示例流程

### 13.2 后端规范

- 路由层只负责路由注册
- API 层只负责参数接收、调用服务、返回响应
- Service 层负责业务逻辑和数据处理
- Model 层负责结构定义
- 不在 Service 中拼装 HTTP 响应
- 新增保护接口时，同步补 API 种子和权限规则

### 13.3 前端规范

- 页面结构尽量保持统一
- 请求统一走现有请求封装
- 页面状态、接口调用、展示逻辑职责清晰
- 演示页也要具备基本可读性，不写一次性堆砌代码
- 复用能力优先抽象成可配置形式，而不是写死单一业务

### 13.4 配置规范

- `server/config.yaml` 只放可提交的安全默认值
- `server/config.debug.yaml` 只用于本地开发，不提交
- 不提交真实数据库、Redis、账号、令牌、密钥

## 14. 菜单、API、权限、种子数据规范

在 BoyKing Admin 当前架构下，新增实验室功能时，除了页面和接口本身，还必须考虑初始化能力。

通常需要同步检查这些点：

- 菜单是否需要在新库中自动生成
- 角色是否需要默认绑定该菜单
- API 是否需要在种子数据中注册
- Casbin 是否需要补默认规则

因此，新增功能时要关注：

- `server/source/system/menu.go`
- `server/source/system/authorities_menus.go`
- `server/source/system/` 下对应 API 和 Casbin 种子文件

原则是：

- Fresh install 能看到基础入口
- 已有数据库升级后，也能自动补齐缺失菜单和权限

## 15. 模块开发流程

建议所有成员按下面流程开发：

1. 先确认功能归属
2. 说明开发背景和目标
3. 先做最小可运行版本
4. 补齐菜单、接口、权限、种子数据
5. 前后端联调
6. 补说明文档
7. 完成构建验证后再提交

如果是复用组件，建议额外补这几项：

- 使用场景
- 接入方式
- 可配置项
- 已知限制
- 搬运到内网时需要同步哪些文件或依赖

## 16. 推荐提交流程

建议采用下面的协作节奏：

1. 先在 issue、群内或口头确认模块目标
2. 小步提交，不做超大混合改动
3. 一个提交尽量只解决一类问题
4. 提交前至少保证构建通过
5. 合并前说明该模块属于哪个分区、解决什么问题、后续如何复用

提交信息建议简洁明确，例如：

- `feat: add customer approval simulation page`
- `feat: add reusable excel error export`
- `refactor: reorganize lab reusable services`
- `docs: add admin-lab project handbook`

## 17. 验证要求

完成开发后，至少进行与改动范围对应的验证。

默认建议：

- 后端执行 `go build -buildvcs=false ./core ./initialize ./service ./service/system ./source/system ./api/v1/... ./router/...`
- 前端执行 `npm run build`

如果功能涉及实际交互，建议再补充：

- 页面手工验证
- 菜单显示验证
- 权限验证
- 导入导出文件实际验证

## 18. 文档要求

`admin-lab` 的价值不仅来自代码，也来自沉淀。

因此，凡是具备长期价值的模块，建议至少补以下文档信息：

- 功能要解决什么问题
- 功能属于哪个分区
- 功能怎么使用
- 功能有哪些关键配置
- 功能迁移到内网时要注意什么

如果是失败实验，也可以保留简短记录，说明为什么放弃，避免团队重复踩坑。

## 19. 内网搬运规范

`admin-lab` 的很多能力最终会搬运到公司内网项目，因此开发时要提前考虑迁移。

建议遵守这些原则：

- 尽量降低和实验环境特有代码的耦合
- 不把页面写死在单一业务字段上
- 依赖引入尽量可控
- 接口契约尽量清晰稳定
- 能抽象的能力优先抽象

搬运时重点检查：

- 前端页面和组件依赖
- 后端 service 与 model 依赖
- 菜单和权限数据
- 字典、参数、配置依赖
- 第三方库是否允许内网环境使用

## 20. 新成员上手建议

新成员进入项目后，建议按下面顺序熟悉：

1. 阅读本手册
2. 阅读 `README.md`
3. 了解实验室三大分区定位
4. 跑起前后端环境
5. 先从已有模块阅读，例如 Excel 实验面板
6. 再开始新增自己的实验模块

## 21. 当前阶段建议

结合项目当前状态，近期建设建议如下：

- 持续完善 `复用组件`
- 优先沉淀 Excel、可视化、通用表格、通用搜索、上传附件等高频能力
- `组件示例` 保持轻量、小而清晰
- `需求仿真` 聚焦真实业务预演，不做无关堆积

## 22. 最终原则

所有成员在开发 `admin-lab` 时，统一遵守下面这条判断标准：

这个功能是否能帮助团队更快验证方案、更稳落地真实需求、更高效复用成果。

如果答案是肯定的，就值得进入 `admin-lab`。
如果答案是否定的，就不必让项目变重。
