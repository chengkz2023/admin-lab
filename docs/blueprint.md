# admin-lab 建设蓝图

## 1. 项目目标

`admin-lab` 的目标是形成一套适合后台研发团队内部复用的“预研资产库”。

输出物应当包括：

- 可直接运行的实验模块
- 可抽取的前端组件
- 可迁移的后端实现
- 搬运说明与接入说明
- 已验证的技术选型结论

## 2. 模块分层建议

建议把能力分成三层：

### 基础能力层

面向多个业务复用的能力：

- 导入导出
- 上传下载
- 通用表格
- 通用筛选器
- 枚举与字典渲染
- 权限控制包装
- 图表容器

### 页面样例层

面向真实业务组合场景：

- 列表页
- 详情页
- 编辑页
- 分步表单
- 可视化分析页
- 审批与状态流页面

### 迁移沉淀层

用于支持内网搬运：

- 接入说明
- 依赖清单
- 需要同步的后端接口
- 菜单权限种子数据说明
- 已知限制与替换点

## 3. 模块准入标准

每个进入 `admin-lab` 的实验模块，尽量满足以下标准：

1. 有明确的实验目标
2. 有最小可运行页面或接口
3. 有可复用的核心实现
4. 有搬运说明
5. 有失败结论时也要记录原因，避免重复踩坑

## 4. 推荐首批落地模块

### 导入导出模块

建议覆盖：

- Excel 导入
- Excel 导出
- 模板下载
- 导入错误提示
- 导入结果回显

### 可视化模块

建议覆盖：

- 折线图
- 柱状图
- 饼图
- 混合图
- 图表筛选联动
- 空数据态处理

### 通用表单模块

建议覆盖：

- 动态字段
- 联动控件
- 校验规则封装
- 只读/编辑双态切换
- 复杂弹窗表单

### 通用表格模块

建议覆盖：

- 可配置列
- 固定列
- 汇总行
- 批量选择
- 行内操作权限
- 大数据量分页体验

## 5. BoyKing Admin 内的落地方式

后端建议按原有链路组织：

`router -> api -> service -> model`

前端建议优先落在：

- `web/src/view/experiment/`
- `web/src/components/lab/`
- `web/src/api/lab/`

这样可以保持实验代码相对集中，后续拆分和迁移更轻松。

## 6. 后续初始化建议

拿到可访问的脚手架后，优先完成：

1. 拉取 BoyKing Admin 基线
2. 将项目名称统一为 `admin-lab`
3. 检查 `server/config.yaml` 是否为安全默认值
4. 增加本地调试专用的 `server/config.debug.yaml` 并加入忽略
5. 新增实验模块总入口，例如“实验室”一级菜单
6. 先落一个完整示例模块，作为后续模板
